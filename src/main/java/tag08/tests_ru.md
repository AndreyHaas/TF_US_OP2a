# Основы софтвер тестирования (Software Testing Basics)

## 1. Что такое тестирование программного обеспечения?

**Тестирование ПО** — это процесс оценки программного продукта с целью выявления различий между существующим и требуемым состоянием (нахождение дефектов) и оценка качества продукта.

> **Ключевая цитата (по ANSI/IEEE 829):** Тестирование — это процесс анализа программного обеспечения с целью обнаружения различий между существующими и требуемыми условиями (дефектами) и оценки характеристик ПО.

## 2. Почему тестирование важно (Цели)?

- **Обнаружение дефектов (багов)** — основная цель.
- **Предотвращение дефектов** (анализ требований до начала разработки).
- **Оценка качества** (насколько продукт готов к релизу).
- **Обеспечение уверенности** в работоспособности продукта.
- **Снижение рисков** и затрат на исправление (дешевле фиксить на ранних этапах).

## 3. Уровни тестирования (Testing Levels)

Тестирование проходит параллельно с разработкой и делится на 4 основных уровня:

| Уровень | Что проверяется | Кто обычно делает |
| :--- | :--- | :--- |
| **Модульное (Unit)** | Отдельные функции/методы/классы | Разработчик |
| **Интеграционное (Integration)** | Взаимодействие между модулями | Разработчик / Тестировщик |
| **Системное (System)** | Вся система целиком (end-to-end) | Тестировщик |
| **Приемочное (Acceptance)** | Соответствие требованиям бизнеса | Заказчик / QA |

## 4. Типы (Виды) тестирования

### По знанию системы:
- **Black Box (Черный ящик):** Не знаем внутреннее устройство, проверяем входы и выходы.
- **White Box (Белый ящик):** Знаем код, проверяем логику, маршруты выполнения.
- **Grey Box (Серый ящик):** Комбинация (знаем структуру БД, но не весь код).

### По времени выполнения:
- **Smoke Testing:** Быстрая проверка критических функций («тестирование сборки»).
- **Sanity Testing:** Проверка того, что исправления багов не сломали логику.
- **Регрессионное (Regression):** Перепроверка старых функций после изменений (самый частый вид).
- **Ретест (Re-test):** Проверка конкретного исправленного бага.

### По атрибуту качества:
- **Функциональное:** Работает ли так, как указано в требованиях?
- **Нагрузочное (Performance):** Скорость, время отклика.
- **Надежности (Stability):** Падает ли система под нагрузкой?
- **Юзабилити (Usability):** Удобно ли пользователю?

## 5. Жизненный цикл бага (Bug Life Cycle)

1.  **New** (Найден) -> 2. **Assigned** (Назначен разработчику)
3.  **Open** (Принят к работе) -> 4. **Fixed** (Исправлен)
5.  **Retest** (Перепроверка тестировщиком)
    - *Если исправлено:* -> **Closed**
    - *Если не исправлено:* -> **Reopened** (обратно к разработчику)

**Иногда:** *Rejected* (отклонен — не баг), *Deferred* (отложен), *Duplicate* (дубликат).

## 6. Тестовая документация (Ключевые артефакты)

1.  **Тест-план (Test Plan):** Что, когда, кто и как будет тестировать.
2.  **Тест-кейс (Test Case):** Шаги для проверки конкретной функции.
    *Состоит из:* ID, Название, Предусловия, Шаги, Ожидаемый результат, Фактический результат.
3.  **Чек-лист (Checklist):** Список того, что нужно проверить (без детальных шагов).
4.  **Баг-репорт (Bug Report):** Сообщение о дефекте.

## 7. Пример структуры баг-репорта

```
Заголовок: [Краткое описание проблемы]
Приоритет: (High / Medium / Low)
Серьезность: (Blocker / Critical / Major / Minor / Trivial)
Окружение: (ОС, Браузер, Версия приложения)
Шаги воспроизведения:
1. Открыть страницу логина
2. Ввести пустой пароль
   Ожидаемый результат: Сообщение "Пароль не может быть пустым"
   Фактический результат: Ошибка сервера 500
   Вложения: (Скриншот, видео, лог ошибки)
```

## 8. Принципы тестирования (7 ключевых)

1.  **Исчерпывающее тестирование невозможно.** (Нельзя проверить ВСЁ).
2.  **Дефекты имеют склонность к кластеризации.** (80% багов скрыто в 20% модулей).
3.  **Тестирование показывает наличие дефектов, но не доказывает их отсутствие.**
4.  **Эффект "Pesticide Paradox":** Если повторять одни и те же тесты, они перестают находить новые баги.
5.  **Раннее тестирование** (начинайте тестировать как можно раньше).
6.  **Контекст зависит** (разный подход для банковского софта и игры).
7.  **Заблуждение об отсутствии ошибок** (система с 0 багов может быть бесполезной для пользователя).

## 9. Роли в команде (Кратко)

- **QA Engineer (Инженер по качеству):** Отвечает за процессы качества (упреждение багов).
- **QC Engineer (Тестировщик):** Непосредственно находит баги.
- **Test Analyst:** Анализирует требования и пишет тест-кейсы.
- **Automation Engineer:** Пишет автотесты.

## 10. Заключение

Тестирование — это **не просто "нажимание кнопок"**. Это дисциплина, требующая аналитического мышления, внимания к деталям и понимания системного подхода. Сегодня мы заложили фундамент.

> **Помните:** Хороший тестировщик не тот, кто нашел много багов, а тот, кто помог команде сделать качественный продукт за минимальное время.
